## 1. Framework Overview

### 1.1 What is BMAD Method?

BMAD Method (**B**uild **M**ore **A**rchitect **D**reams) is an npm package that installs an AI-driven agile development framework into project directories. Unlike traditional AI tools that generate code autonomously, BMAD agents and workflows act as expert collaborators that guide users through structured processes, combining human insight with AI capability.

The framework provides:

- **CLI tooling** for installation and management (`npx bmad-method install`)
- **Specialized AI agents** (PM, Architect, Developer, UX, Analyst, Tech Writer) with declarative YAML definitions
- **Structured workflows** organized in four phases: Analysis, Planning, Solutioning, Implementation
- **Module ecosystem** supporting official modules (BMM, BMB, TEA, BMGD, CIS) and custom extensions
- **IDE integration** for 15+ AI IDEs including Claude Code, Cursor, GitHub Copilot, Codex CLI
- **Intelligent help system** (`bmad-help`) that inspects your project and recommends next steps

### 1.2 Core Philosophy

BMAD Method is built on the principle of **Human Amplification** — the AI creates conditions for the user's insight rather than manufacturing answers itself. Key philosophical tenets:

- **Orient-First Pattern**: Agents must read existing context before generating new content
- **Micro-File Architecture**: Complex logic is sharded into small, single-purpose files to keep LLM context windows clean
- **Progressive Context Building**: Each phase produces documents that inform the next, creating a chain of validated artifacts
- **Structured Agility**: Workflow steps provide guardrails while allowing creative freedom within boundaries
- **Skills-Based Architecture**: Everything is a skill — agents, workflows, and tasks are unified under a single format

### 1.3 Key Concepts

| Concept | Definition |
|---------|------------|
| **Agent** | A specialized AI persona (e.g., Architect, PM) defined by a YAML manifest with role, style, and menu triggers |
| **Skill** | The primary unit of functionality — a directory containing a `SKILL.md` entrypoint and supporting logic |
| **Workflow** | A structured, multi-step process designed to achieve a specific development outcome |
| **Phase** | One of the four stages of the BMad development lifecycle: Analysis, Planning, Solutioning, or Implementation |
| **Artifact** | A file generated by a workflow, stored in `_bmad-output/` |
| **Sidecar** | Persistent memory/instructions associated with a specific agent to maintain context across sessions |
| **Module** | A self-contained package of agents, workflows, and skills that extends BMAD capabilities |
| **Distillate** | A lossless, LLM-optimized compression of a document to preserve context window space |
| **Shard** | A small, focused file produced by splitting large documents for better LLM consumption |
| **Release Channel** | Distribution stream: `stable` for general use, `next` for bleeding-edge features |

## 2. Architecture

### 2.1 System Architecture Overview

BMAD Method operates through five primary subsystems:

| Subsystem | Purpose |
|-----------|---------|
| **Installation System** | Orchestrates module installation, configuration, and IDE setup |
| **Module System** | Manages official and external modules with release channels |
| **Manifest System** | Generates searchable CSV metadata for agents and skills |
| **Customization Engine** | Merges TOML-based overrides for agents and workflows |
| **IDE Integration** | Generates platform-specific skill files for 20+ IDEs |

### 2.2 Installation Flow

The installation process transforms source definitions into executable IDE artifacts:

1. User runs `npx bmad-method install`
2. CLI invokes the Installer class which orchestrates the process
3. InstallPaths creates directory structure
4. ExistingInstall detects prior installations
5. Config.build() gathers user preferences interactively
6. OfficialModules and ExternalModuleManager copy skill files
7. SkillScanner discovers all SKILL.md files
8. IdeManager generates platform-specific files (`.claude/skills/`, `.cursor/skills/`, etc.)
9. ManifestGenerator produces CSV metadata

### 2.3 Directory Structure

The installation creates the following structure:

```
_bmad/
├── core/                 # Core module skills
│   └── bmad-help/        # Intelligent help
├── bmm/                  # BMad Method module
│   ├── agents/           # Specialized agent definitions
│   └── workflows/        # Sharded workflow directories
├── config.toml           # Central TOML configuration (installer-owned)
├── config.user.toml      # User-scoped config (installer-owned)
├── custom/               # User overrides (TOML)
│   ├── config.toml       # Team overrides (committed to git)
│   ├── config.user.toml  # Personal overrides (gitignored)
│   └── {skill-name}.toml # Per-skill team overrides
└── _memory/              # Agent sidecars (runtime persistence)
    └── *-sidecar/

_bmad-output/
├── planning-artifacts/     # Phase 1-3 outputs (PRD, Architecture, Epics)
└── implementation-artifacts/ # Phase 4 outputs (Sprints, Stories, Reviews)
```

### 2.4 Skills-Based Architecture

In v6, everything is a skill. A skill is a directory containing a `SKILL.md` entrypoint. Skills follow strict validation rules:

- **SKILL-04**: Name must start with `bmad-`, lowercase, hyphens only
- **SKILL-05**: The `name` in frontmatter must match the directory name
- **SKILL-07**: `SKILL.md` must have body content providing instructions

### 2.5 Micro-File Architecture

The structural philosophy uses "shards" — small, single-purpose files:

- **Sharded Workflows**: Breaking complex logic into multiple step files (`step-01-init.md`, `step-02-elicit.md`, etc.)
- **Sharded Artifacts**: Large documents (like PRDs) are sharded into smaller pieces for focused editing
- **Sidecars**: Persistent agent-specific memory stored alongside agent definitions

### 2.6 Manifest System

The `ManifestGenerator` bridges the file system and AI context:

- `collectSkills()` recursively walks module directories to find `SKILL.md` files
- Canonical IDs derive from directory names
- `skill-manifest.csv` provides searchable metadata for agents and skills
- `files-manifest.csv` tracks SHA256 hashes of installed files to detect user modifications
- Agent descriptors from `module.yaml` feed into `config.toml` for the unified roster

## 3. Agents

### 3.1 Agent Architecture

Agents are role-based AI personas with specialized expertise, communication styles, and menu-driven access to workflows. Each agent is defined in `*.agent.yaml` (or compiled `.md`) files with:

| Component | Purpose |
|-----------|---------|
| **Identity** | `persona.role`, `persona.identity` — defines who and what the agent is |
| **Behavior** | `persona.principles` — guiding rules for decision making |
| **Interface** | `menu` — array of triggers mapping to workflow names/skills |
| **Team** | `team` — unified grouping for retrospective and party-mode skills |

### 3.2 Agent Activation (8-Step Process)

When an agent is invoked, it follows this exact activation sequence:

1. **Resolve the agent block**: Merge default `customize.toml` with team and personal overrides
2. **Execute prepend steps**: Pre-flight behaviors configured by the team
3. **Adopt persona**: Agent adopts its hardcoded identity and layers on customized attributes
4. **Load persistent facts**: Foundational context carried throughout the session
5. **Load config**: User name, communication language, output language, artifact paths
6. **Greet**: Personalized greeting in the configured language, prefixed with agent's emoji icon
7. **Execute append steps**: Post-greet setup configured by the team
8. **Dispatch or present the menu**: If command provided, execute it; otherwise show numbered menu

### 3.3 Agent Compilation Process

Agents are compiled from YAML source definitions into executable Markdown instructions:

```
Source: pm.agent.yaml + TOML overrides
   ↓
Merge: TOML Override Merge (structural rules)
   ↓
Resolve: Variable Resolver ({project-root}, {planning_artifacts})
   ↓
Manifest: ManifestGenerator → config.toml generation
   ↓
Output: _bmad/bmm/agents/pm.md + IDE skill command
```

### 3.9 Agent Interaction Patterns

Agents interact through several patterns:

- **Menu Dispatch**: User selects a menu code, agent loads the corresponding workflow
- **Party Mode**: Multiple agents spawned as subagents for multi-perspective discussions
- **Sequential Workflow**: Agents hand off artifacts to each other across phases (e.g., PM creates PRD → Architect uses PRD for architecture)
- **Workflow Handlers**: Menu items can reference `exec` (execute file), `workflow` (start workflow), or conversational mode

### 3.10 Trigger Types

Agent menu triggers use two invocation types:

- **Workflow triggers** (no arguments needed): Load a structured workflow file. Examples: `CP` (Create PRD), `DS` (Dev Story), `CA` (Create Architecture)
- **Conversational triggers** (arguments required): Start a free-form conversation. Examples: `WD` (Write Document — provide description), `MG` (Mermaid Generate — provide diagram description)

### 4.4 Workflow Execution Model

#### Sharded Step Architecture

Workflows are broken into step files following the pattern `step-XX-slug.md`:

- Each step contains a `HALT` command to prevent the LLM from over-executing
- Each step focuses on a single logical task
- Steps update project artifacts in `_bmad-output/`
- Step filenames must follow `step-XX-name.md` format (validator rule STEP-01)
- Workflows must have 2-10 steps (validator rule STEP-07)

#### Workflow Frontmatter

Workflows use YAML frontmatter for state tracking:
- `stepsCompleted`: Tracks which steps have been executed
- `status`: Current workflow status (e.g., `in-progress`, `complete`)
- Configuration variables like `{project-root}`, `{planning_artifacts}` are resolved at runtime

#### Orient-First Pattern

When a workflow skill is invoked, the agent first inspects the project state (via `bmad-help` or direct file inspection) before generating content. This ensures:
- Existing artifacts are discovered and loaded
- Context from previous phases informs current work
- LLM doesn't produce generic or contradictory content

#### Artifact Discovery Protocol (discover_inputs)

Workflows use a dual discovery system:
1. Try loading the whole document (e.g., `PRD.md`)
2. If not found, check for sharded folder (e.g., `prd/index.md`)

## 5. Models/LLM Configuration

### 5.1 Configuration Approach

BMAD Method is **model-agnostic** — it does not prescribe specific LLM models or manage API keys. Instead, it generates prompt-based artifacts that work with whatever AI IDE the user has configured. The framework's output is IDE-specific skill directories (`.claude/skills/`, `.cursor/skills/`, etc.) that surface as commands in the user's chosen AI tool.

### 5.2 Model Selection in Party Mode

The only explicit model configuration occurs in Party Mode, where a `--model` argument can force all subagents to use a specific model (e.g., `haiku` for lightweight tasks, `opus` for deep reasoning). The choice of model matches the depth of thinking required.

### 5.3 config.yaml Structure

The BMM module's `config.yaml` (generated during installation) contains:

```yaml
user_skill_level: intermediate          # beginner | intermediate | expert
planning_artifacts: "{project-root}/_bmad-output/planning-artifacts"
implementation_artifacts: "{project-root}/_bmad-output/implementation-artifacts"
project_knowledge: "{project-root}/docs"
user_name: ""
communication_language: English
document_output_language: English
output_folder: "{project-root}/_bmad-output"
```

Key config variables:
- `user_skill_level`: Affects how agents explain concepts (beginner gets more explanation, expert is direct)
- `planning_artifacts`: Path for Phase 1-3 outputs
- `implementation_artifacts`: Path for Phase 4 outputs
- `project_knowledge`: Path for long-lived knowledge artifacts
- `communication_language`: Language agents use in chat
- `document_output_language`: Language for generated documents

### 5.4 Context Window Management

The framework employs several strategies to manage LLM context:

- **Distillation**: Lossless compression into dense bullet format
- **Sharding**: Splitting large documents into manageable chunks
- **Orient-First**: Loading only relevant context before generating
- **Sidecars**: Persistent agent memory loaded on demand, not all at once
- **Index Files**: Quick-scanning directory indexes instead of full file reads

## 6. Modules

### 6.1 Module Architecture

Modules are the primary extensibility unit. Each module consists of:

- `module.yaml`: Defines code, name, description, configuration prompts, agents, and directories
- `module-help.csv`: Catalog of skills with menu codes, phases, dependencies, and output locations
- `agents/`: Agent definition files
- `workflows/`: Workflow step files organized by phase
- `config.yaml`: Generated configuration from installer answers
- `customize.toml`: Default customization surface

### 6.2 Module Installation

Modules are installed via the unified CLI:

```bash
npx bmad-method install
```

The installer:
- Presents interactive prompts for each module's configuration
- Supports `stable` and `next` release channels per module
- Fetches external modules from GitHub with CDN fallback
- Generates IDE-specific skill files

## 8. Teams and Party Mode

### 8.1 Team System

Agents are organized into teams via the `team` field in their `module.yaml` descriptor. Teams enable:

- **Grouping agents** for Party Mode selection
- **Filtering** for retrospectives and roundtable discussions
- **Cross-module collaboration** (e.g., combining BMM and CIS agents)

Default team: `software-development` (all BMM agents)

Custom teams can be defined in `_bmad/custom/config.toml`:

```toml
[agents.kirk]
team = "startrek"
name = "Captain James T. Kirk"
title = "Starship Captain"
icon = "🖖"
description = "Bold, rule-bending commander."
```

### 8.2 Party Mode (`bmad-party-mode`)

Party Mode orchestrates multi-agent group discussions where multiple specialized agents collaborate in a single conversation.

#### How It Works

1. **Roster Resolution**: Calls `resolve_config.py` to identify all installed agents from merged configuration layers
2. **Voice Selection**: Based on the user's query, selects 2-4 relevant agents
3. **Context Injection**: Each subagent receives a specialized prompt with its persona, conversation summary, and relevant `project-context.md` snippets
4. **Parallel Execution**: Agents are spawned in parallel to ensure independent thinking and avoid "opinion convergence"

#### Modes of Operation

- **Subagent Mode (Default)**: Spawns independent processes for authentic diversity of thought
- **Solo Mode (`--solo`)**: Orchestrator roleplays all agents in a single response (for speed or when subagent tools are unavailable)

#### When to Use

- High-stakes technical decisions (ADRs)
- Post-mortems and retrospectives
- Cross-domain requirement analysis
- Seeking diverse perspectives on complex decisions

## 9. Configuration System

### 9.1 Four-File Configuration Architecture

BMAD uses a tiered configuration system with clean separation between system defaults, user overrides, and custom content:

```
Priority 1 (wins): _bmad/custom/config.user.toml     (personal, gitignored)
Priority 2:        _bmad/custom/config.toml           (team overrides, committed)
Priority 3:        _bmad/config.user.toml              (installer-owned, user scope)
Priority 4 (base): _bmad/config.toml                  (installer-owned, team scope)
```

### 9.2 Per-Skill Customization (Three-Layer Override)

For individual agents or workflows:

```
Priority 1 (wins): _bmad/custom/{skill-name}.user.toml  (personal, gitignored)
Priority 2:        _bmad/custom/{skill-name}.toml        (team/org, committed)
Priority 3 (last): {skill-root}/customize.toml            (defaults shipped with skill)
```

Override files are **sparse** — include only the fields being changed.

### 9.3 Merge Rules

The resolver applies four structural rules based on data shape:

| Shape | Rule |
|-------|------|
| Scalar (string, int, bool, float) | Override wins |
| Table | Deep merge (recursively apply rules) |
| Array of tables with `code` or `id` keys | Merge by key — matching keys replace, new keys append |
| Any other array | Append — base items first, then team, then user |

**No removal mechanism** — overrides cannot delete base items. To suppress a default, override by `code` with a no-op.

### 9.4 Customizable Fields

#### Agent Customization (under `[agent]`)

- `icon`: Emoji icon
- `role`: Agent's role description
- `communication_style`: How the agent speaks
- `principles`: Append-only array of guiding principles
- `persistent_facts`: Append-only array of static facts or `file:` references
- `activation_steps_prepend`: Steps to run before persona adoption
- `activation_steps_append`: Steps to run after greeting
- `[[agent.menu]]`: Array of tables, merged by `code` field

#### Workflow Customization (under `[workflow]`)

- `activation_steps_prepend`: Pre-flight context loads
- `activation_steps_append`: Post-greeting setup
- `persistent_facts`: Foundational context for the workflow run
- `on_complete`: Action to run when workflow finishes

### 9.5 Central Configuration Sections

- `[core]` and `[modules.<code>]`: Install answers
- `[agents.<code>]`: Agent essence (code, name, title, icon, description, team)

### 9.6 Editing Rules

- `_bmad/config.toml` and `_bmad/config.user.toml` are **regenerated every install** — treat as read-only
- `_bmad/custom/config.toml` and `_bmad/custom/config.user.toml` are **never touched by the installer** — this is the correct surface for customizations
- Copying full `customize.toml` into an override is **actively harmful** — it locks in old defaults

## 10. Knowledge System

### 10.1 project-context.md

The `project-context.md` file acts as a **constitution** for AI agents, ensuring consistent code generation across sessions.

**Location**: `_bmad-output/project-context.md`

**What it contains**:
- **Technology Stack and Versions**: Frameworks, languages, and tools with specific versions
- **Critical Implementation Rules**: Unobvious patterns, conventions, TypeScript config, code organization, testing patterns

**How agents use it**:
- Every implementation workflow automatically loads it if it exists
- Content is injected as "persistent facts" into agent context
- Influences decisions in `bmad-create-architecture`, `bmad-create-story`, `bmad-dev-story`, `bmad-code-review`
- Can be created manually or generated via the `bmad-generate-project-context` workflow

### 10.2 Persistent Facts

Static facts that agents keep in mind for the entire session. Can be:
- **Literal sentences**: "Our org is AWS-only — do not propose GCP or Azure."
- **File references**: `file:{project-root}/docs/compliance/hipaa-overview.md` (glob patterns supported)

Facts are configured in two locations:
- Per-agent: `_bmad/custom/{skill-name}.toml` under `[agent].persistent_facts`
- Per-workflow: `_bmad/custom/{workflow}.toml` under `[workflow].persistent_facts`

### 10.3 Agent Sidecars

Agent sidecars are persistent memory files stored in `_bmad/_memory/*-sidecar/`. They maintain context across sessions, such as:
- `memories.md`: Running memory of decisions and preferences
- `instructions.md`: Agent-specific instructions
- `documentation-standards.md`: Tech Writer's accumulated documentation conventions

### 10.4 Distillation

The `bmad-distillator` skill produces lossless, LLM-optimized compressions:

- Transforms prose into dense bullet format
- YAML frontmatter includes `type: bmad-distillate`, `sources`, `downstream_consumer`, `token_estimate`
- Deduplicates facts across multiple source documents
- Converts decision prose into "Rejected: [Option]. Reason: [Rationale]" patterns
- Verifiable via round-trip reconstruction

### 10.5 Seed Knowledge

BMB includes seed knowledge fragments in Markdown files that serve as a centralized knowledge base:

- Agent design patterns and architectures
- Menu design patterns
- Persona crafting properties
- Principles for agent design
- Compilation process documentation

These are loaded as persistent facts during BMB workflow execution.

## 11. Best Practices

### 11.1 Using the Framework Effectively

1. **Start with `bmad-help`**: Always invoke `bmad-help` first — it analyzes your project state and recommends the next step
2. **Follow the Phase Order**: The four phases build progressively; skipping creates gaps in context
3. **Use Quick Flow for Simple Tasks**: Don't over-engineer simple bug fixes — use `QD` (Quick Dev) or `QS` (Quick Spec)
4. **Generate project-context.md Early**: For brownfield projects, run `GPC` first to give agents project-specific knowledge
5. **Validate Before Moving On**: Run `VP` (Validate PRD), `IR` (Implementation Readiness) before proceeding to the next phase
6. **Shard Large Documents**: Use `bmad-shard-doc` when documents exceed 500 lines
7. **Use Party Mode for Decisions**: Get multiple agent perspectives on high-stakes decisions
8. **Customize Sparingly**: Only override fields you need to change; full copies lock in old defaults

### 11.2 Brownfield Projects

For existing/legacy codebases:

1. Run `bmad-generate-project-context` (GPC) to scan the codebase
2. Run `bmad-document-project` (DP) to produce documentation
3. Use the generated `project-context.md` to align agents with existing patterns
4. Start from the phase that matches current project state (not necessarily Phase 1)

### 11.3 Context Management

- Use `bmad-distillator` when context windows are tight
- Use `bmad-shard-doc` for documents over 500 lines
- Use `bmad-index-docs` for folder-level navigation
- Use `bmad-advanced-elicitation` to refine and improve generated content

### 11.4 Quality Assurance

- Use `bmad-review-adversarial-general` for critical document reviews
- Use `bmad-review-edge-case-hunter` alongside adversarial review for orthogonal coverage
- Run editorial structure review **before** prose review
- Use Checkpoint Review (`CK`) for human-in-the-loop code review

### 11.5 Preventing Agent Conflicts

To prevent agents from making contradictory decisions:

- Maintain comprehensive architecture documentation
- Use Architecture Decision Records (ADRs) as shared context
- Keep `project-context.md` updated with current conventions
- Use Party Mode to surface conflicting assumptions
